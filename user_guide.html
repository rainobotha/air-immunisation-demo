1. Introduction & Objectives

This demo showcases a complete, production-style data pipeline built entirely on Snowflake — from raw file ingestion through to an interactive analytics dashboard — using the Australian Immunisation Register (AIR) as the use case.

The objectives of this demo are to illustrate:

  • Simplicity — The entire pipeline is pure SQL. No Spark, no PySpark notebooks, no cluster configuration, no Airflow DAGs.
  • Zero orchestration — Snowflake Dynamic Tables handle all pipeline scheduling and incremental refresh automatically.
  • Cost effectiveness — Everything runs on an X-Small warehouse (~$0.50 total) with 60-second auto-suspend.
  • Integrated analytics — The Streamlit dashboard runs inside Snowflake. No separate BI tool, no app server, no additional infrastructure.
  • Data quality handling — The pipeline demonstrates real-world data quality challenges (malformed dates, duplicates, missing values) and how they are resolved declaratively.
Target Audience Technical decision-makers evaluating Snowflake against Databricks. The demo is designed to be self-service — clone the repo, follow the steps, and see results in under 30 minutes.

2. The Australian Immunisation Register

The Australian Immunisation Register (AIR) is a national register that records all vaccinations given to people of all ages in Australia. It is managed by Services Australia on behalf of the Department of Health and Aged Care.

What the AIR captures

Vaccination providers (GPs, pharmacies, hospitals, community health centres, Aboriginal Medical Services, and school programs) are required by law to report all vaccinations administered to the AIR within 10 working days. The data captured includes:

CategoryData Elements
Patient identityMedicare number, first name, last name, date of birth, gender
DemographicsState/territory, postcode, address, Indigenous status
Vaccination detailsVaccine brand, antigen(s), dose number, batch number, vial serial number
AdministrationDate administered, date reported, administration site (left arm, right thigh, etc.), route (IM, SC, oral)
ProviderProvider number, practice name, provider type, state
FundingNIP-funded (Y/N), school program (Y/N)

National Immunisation Program (NIP) Schedule

The NIP schedule defines the vaccines provided free of charge to eligible Australians. For children, this includes vaccinations at birth, 2, 4, 6, 12, and 18 months, and at 4 years of age. Key vaccines include:

Adult vaccinations in the dataset include COVID-19 (Comirnaty, Spikevax), influenza (Fluarix Tetra), HPV (Gardasil 9), pertussis boosters (Boostrix), shingles (Shingrix), and pneumococcal (Pneumovax 23).

5. Pipeline Detail: Bronze Layer

The bronze layer preserves source fidelity — all columns are stored as STRING type exactly as received. No transformations, no parsing, no data loss. This is the immutable record of what was received from the source system.

Key design decisions

  • All STRING columns — Dates remain as strings ("2026-04-14", "14/04/2026", etc.) so no data is lost during ingestion. Parsing happens in the silver layer.
  • Metadata columns — _loaded_at (TIMESTAMP) captures when each record was ingested.
  • ON_ERROR = 'CONTINUE' — Ingestion continues past malformed rows rather than failing the entire load.
  • PATTERN matching — COPY INTO uses file name patterns (.*patients.*) so files can be uploaded in any order.

Technology used

ComponentSnowflake Feature
File uploadInternal Stage (AIR_LANDING) — upload via Snowsight UI
File formatNamed File Format (CSV_FORMAT) — reusable, version-controlled
Data loadingCOPY INTO — bulk load with pattern matching

6. Pipeline Detail: Silver Layer

The silver layer cleanses, conforms, deduplicates, and enriches the raw data. All four silver tables are Dynamic Tables — they are defined as declarative SQL queries and Snowflake automatically keeps them up to date.

What are Dynamic Tables? Dynamic Tables are Snowflake's declarative data pipeline feature. You write a SQL query that defines the target state of the table, set a TARGET_LAG (e.g. 1 hour), and Snowflake handles all the scheduling, dependency tracking, and incremental refresh automatically. No Airflow. No cron jobs. No Spark.

The main enriched fact table — joins vaccinations with patient demographics and provider details. This single table powers most of the gold layer and dashboard queries. Records with unparseable administration dates are excluded.

7. Pipeline Detail: Gold Layer

The gold layer contains analytics-ready aggregates — pre-computed metrics that the Streamlit dashboard queries directly. All five tables are Dynamic Tables with a 1-hour target lag.

GOLD.COVERAGE_BY_STATE_AGE

The core immunisation coverage metric. Calculates the percentage of patients who have received 3+ distinct antigens, broken down by state, age group, and Indigenous status. This mirrors how the real AIR reports national coverage statistics.

GOLD.MONTHLY_VACCINATION_TRENDS

Monthly time series showing dose volumes, unique patients vaccinated, active provider counts, and average reporting lag — grouped by state, antigen, and NIP funding status.

GOLD.PROVIDER_PERFORMANCE

Per-provider activity metrics: total doses administered, unique patients served, vaccine diversity, reporting timeliness (average lag, late report count), and NIP-funded percentage.

GOLD.DATA_QUALITY_SUMMARY

Pipeline health metrics showing date parse failure counts and percentages for both patients and vaccinations — demonstrating the value of the bronze→silver cleansing pipeline.

GOLD.CHILDHOOD_COVERAGE_MILESTONES

Coverage rates at the key AIR reporting milestones: 1 year, 2 years, and 5 years of age. Compares against the national 95% aspirational target and calculates the gap, broken down by state and Indigenous status.

9. Deployment Guide

No installs required Everything is done through the Snowsight web UI. No CLI, no Python, no batch files. Just a browser and a Snowflake account.

Estimated time: 20-30 minutes (mostly waiting for file uploads and Dynamic Table refreshes).

1

Setup environment & bronze tables

  1. Log into Snowsight
  2. Open Projects > Worksheets and create a new SQL worksheet
  3. Open scripts/01_setup_bronze.sql from the repository and paste the entire contents into the worksheet
  4. Click Run All (Ctrl+Shift+Enter)
  5. Stop when you reach the "STOP HERE" comment — you need to upload files next

This creates: warehouse (AIR_DEMO_WH), database (AIR_DEMO), four schemas, file format, stage, and three bronze tables.

2

Upload CSV files to stage

  1. In the left sidebar, navigate to Data > Databases > AIR_DEMO > BRONZE > Stages > AIR_LANDING
  2. Click the + Files button
  3. Upload all three files from the data/ folder:
    • patients.csv (~18 MB)
    • providers.csv (~275 KB)
    • vaccinations.csv (~170 MB — this one takes a minute)
3

Load data into bronze tables

  1. Go back to your worksheet from Step 1
  2. Select and run the COPY INTO statements (everything after the "STOP HERE" comment)
  3. The validation query at the bottom should show:
    • RAW_PATIENTS: ~100,000 rows
    • RAW_PROVIDERS: ~2,000 rows
    • RAW_VACCINATIONS: ~751,000 rows
4

Create silver Dynamic Tables

  1. Open a new worksheet (or clear the current one)
  2. Paste the contents of scripts/02_silver_dynamic_tables.sql
  3. Click Run All

Creates 4 Dynamic Tables: PATIENTS, PROVIDERS, VACCINATIONS, VACCINATION_EVENTS

5

Create gold Dynamic Tables

  1. Paste the contents of scripts/03_gold_dynamic_tables.sql
  2. Click Run All

Creates 5 analytics tables: coverage, trends, provider performance, data quality, childhood milestones.

6

Trigger initial refresh

  1. Paste the contents of scripts/05_manual_refresh.sql
  2. Click Run All

This forces all 9 Dynamic Tables to compute immediately. The verification query at the bottom shows row counts for every table.

Note If you skip this step, the Dynamic Tables will still refresh automatically — they just might take up to 1 hour (the configured TARGET_LAG).
7

Deploy the Streamlit dashboard

  1. Open a new worksheet
  2. Paste the contents of scripts/04_deploy_streamlit.sql
  3. Run the first part (up to the "STOP HERE" comment) to create the Streamlit stage
  4. Navigate to Data > Databases > AIR_DEMO > APP > Stages > STREAMLIT_STAGE
  5. Click + Files and upload streamlit/streamlit_app.py
  6. Go back to the worksheet and run the CREATE STREAMLIT statement
  7. Navigate to Projects > Streamlit — click AIR_DASHBOARD to open it
Done! The dashboard is live. All data pipelines are automated. Dynamic Tables will continue refreshing on their 1-hour schedule — if new data is loaded into the bronze tables, everything downstream updates automatically.

10. Snowflake vs Databricks

This demo is purpose-built to highlight the operational simplicity of Snowflake compared to a Databricks implementation of the same pipeline.

Capability Snowflake (this demo) Databricks equivalent
Compute ✓ X-Small warehouse, auto-suspend 60s, instant resume ✗ Cluster provisioning, spin-up time (2-5 min), DBU pricing
Data ingestion ✓ COPY INTO from stage — one SQL command ✗ Auto Loader or COPY INTO with Delta Lake configuration
Pipeline orchestration ✓ Dynamic Tables — zero config, declarative SQL ✗ Delta Live Tables + Workflows (or external Airflow)
Storage format ✓ Managed automatically — no format decisions ✗ Delta Lake setup, compaction, Z-ordering
Language ✓ Pure SQL — accessible to any analyst ✗ PySpark notebooks (requires Python/Spark skills)
Dashboard ✓ Streamlit in Snowflake — built-in, no extra cost ✗ Separate BI tool (Tableau, Power BI) or Databricks SQL dashboards
Governance ✓ Built-in RBAC, masking, lineage, column-level security ✗ Unity Catalog setup required (separate configuration)
Deployment ✓ Copy-paste SQL into Snowsight. Done. ✗ Workspace setup, cluster policies, notebook deployment, workflow creation

12. Cleanup

To remove all demo objects, run scripts/99_cleanup.sql or execute:

DROP STREAMLIT IF EXISTS AIR_DEMO.APP.AIR_DASHBOARD;
DROP DATABASE IF EXISTS AIR_DEMO;
DROP WAREHOUSE IF EXISTS AIR_DEMO_WH;

This removes everything — the database, all schemas, all tables, the stage, and the warehouse. Clean slate.
